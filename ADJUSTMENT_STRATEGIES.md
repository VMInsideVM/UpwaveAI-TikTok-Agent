# Agent 筛选条件调整策略文档

## 概述

当找到的达人数量不足时，Agent 会根据数量缺口的严重程度，提供不同的调整建议方案。这个文档详细说明了所有调整策略和优先级。

---

## 数量评估标准

Agent 使用双重标准评估达人数量：

### 计算公式

```python
真实数量（展示给用户）= 最大页数 × 10
保守估计（内部判断用）= 最大页数 × 5
```

### 状态分类

| 状态 | 判断条件 | Agent 行为 |
|------|----------|-----------|
| **充足** | 保守估计 ≥ 用户需求 | 直接进入排序选择，开始搜索 |
| **可接受** | 保守估计 ≥ 用户需求 × 50% | 询问用户是否接受当前数量或调整条件 |
| **严重不足** | 保守估计 < 用户需求 × 50% | 强烈建议调整，提供 3-5 个具体方案 |

**示例**:
- 用户需要 50 个达人
- 最大页数为 30 页
- 真实数量 = 30 × 10 = 300 个（告诉用户）
- 保守估计 = 30 × 5 = 150 个（内部判断）
- 150 ≥ 50 → 状态为**充足**

---

## 调整方案优先级

当达人数量不足时，Agent 会按以下优先级生成调整建议（最多 6 个方案）：

### 优先级 1: 放宽粉丝数范围（效果最好）

**适用条件**: 用户设置了粉丝数范围（`followers_min` 或 `followers_max`）

**调整策略**:

| 缺口程度 | 下限调整 | 上限调整 | 预期增加 |
|---------|---------|---------|---------|
| 缺口很大（需要 3 倍以上） | 减半 | 扩大 3 倍 | 100-150% |
| 缺口适中（需要 1-3 倍） | 降低 30% | 扩大 50% | 50-100% |

**示例**:
```
当前: 10万 - 50万 粉丝
调整后: 5万 - 150万 粉丝（缺口很大时）
或: 7万 - 75万 粉丝（缺口适中时）

理由: 粉丝数是主要限制因素，放宽范围可显著增加候选达人
```

---

### 优先级 2: 放宽商品分类范围（新增功能 ⭐）

**适用条件**: 当前分类是 L2 或 L3 层级

**调整策略**:

| 当前层级 | 调整后层级 | 预期增加 | 说明 |
|---------|----------|---------|------|
| L3（三级分类） | L2（二级分类） | 40-60% | 从精确分类放宽到上一级 |
| L2（二级分类） | L1（一级分类） | 60-100% | 从二级放宽到主分类 |

**示例 1: L3 → L2**
```
当前: L3 分类: 盘
调整后: L2 分类: 餐具

理由: 将分类从三级"盘"放宽到二级"餐具"，可包含更多相关达人
      （如碗、筷子、叉子等餐具类达人也会被包含）
```

**示例 2: L2 → L1**
```
当前: L2 分类: 餐具
调整后: L1 分类: 厨房用品

理由: 将分类从二级"餐具"放宽到一级"厨房用品"，显著扩大候选范围
      （包含厨房器具、饮具、餐具等所有厨房相关达人）
```

**技术实现**:
- 自动读取分类 JSON 文件，找到上一级分类的 ID 和 URL 后缀
- L3 → L2: 在同一分类文件中查找父级 L2
- L2 → L1: 直接使用主分类（L1）ID

**注意事项**:
- 这是**唯一允许修改商品分类**的情况
- 必须通过 `suggest_parameter_adjustments` 工具生成方案
- 用户必须明确选择该方案才能执行
- 如果当前已经是 L1 分类，则不会生成此方案

---

### 优先级 3: 移除新增粉丝数限制

**适用条件**: 用户设置了新增粉丝限制（`new_followers_min` 或 `new_followers_max`）

**调整策略**:
```
当前: 限制在 1万 - 10万
调整后: 无限制

预期增加: 20-30%
理由: 新增粉丝数要求过于严格，移除可包含更多稳定型达人
```

---

### 优先级 3: 移除联盟达人限制

**适用条件**: 用户勾选了"仅联盟达人"（`affiliate_check = True`）

**调整策略**:
```
当前: 仅联盟达人
调整后: 所有达人（包含非联盟）

预期增加: 30-50%
理由: 扩大候选池，包含更多潜在合作达人
```

---

### 优先级 4: 移除认证类型限制

**适用条件**: 用户限制了认证类型（`auth_type = 'verified'` 或 `'not_verified'`）

**调整策略**:
```
当前: 仅已认证达人（或仅未认证达人）
调整后: 不限制认证状态

预期增加: 10-20%
理由: 包含所有认证状态的达人
```

---

### 优先级 5: 移除账号类型限制

**适用条件**: 用户限制了账号类型（`account_type = 'personal'` 或 `'business'`）

**调整策略**:
```
当前: 仅个人账号（或仅企业账号）
调整后: 不限制账号类型

预期增加: 5-15%
理由: 包含个人和企业两种账号类型
```


**用户选项**:
- 如果用户选择"接受" → 进入排序选择

## 调整执行流程

### 用户选择方案后

1. **应用新参数**:
   - Agent 更新 `current_params` 中的相关字段
   - 保持国家和商品分类不变（这是强制规则）

2. **重新构建 URL**:
   - 使用 `build_search_url` 工具，传入新参数
   - 保持商品分类后缀不变

3. **重新检查数量**:
   - 使用 `get_max_page_number` 获取新的最大页数
   - 使用 `analyze_quantity_gap` 重新评估状态

4. **循环判断**:
   - 如果仍然不足 → 继续展示调整建议
   - 如果充足/可接受 → 进入排序选择或询问用户意见

---

## 不可修改的参数

以下参数一旦确定，**绝对不能修改**：

1. **国家/地区** (`country_name`)
   - 原因: 国家决定了数据源和分类体系
   - 如需更改: 必须重新开始新的搜索

2. **商品分类** (`category_info`)
   - 原因: 分类是通过 LLM 语义推理得出的，修改可能导致数据不匹配
   - 如需更改: 必须重新输入商品名称

## 技术实现

### 核心函数

**文件**: [adjustment_helper.py](adjustment_helper.py)

1. **`analyze_quantity_gap(max_pages, user_needs)`**
   - 判断数量是否充足
   - 返回状态和给用户的提示消息

2. **`suggest_adjustments(current_params, target_count, current_count)`**
   - 生成 3-5 个调整方案
   - 按优先级排序
   - 根据缺口程度调整放宽幅度

### Agent 工具

**文件**: [agent_tools.py](agent_tools.py)

- **`AnalyzeQuantityTool`**: 封装 `analyze_quantity_gap` 为 LangChain 工具
- **`SuggestAdjustmentsTool`**: 封装 `suggest_adjustments` 为 LangChain 工具

### Prompt 设计

**文件**: [agent.py](agent.py) - 第 148-184 行

- 详细定义了三种情况的处理流程
- 明确要求 Agent 展示调整方案的格式
- 强调等待用户回复，不自动调整

## 常见问题

### Q1: 为什么不自动应用调整方案？

**A**: 尊重用户意图。筛选条件是用户精心设置的，自动修改可能违背用户需求。

### Q2: 可以组合多个调整方案吗？

**A**: 可以。用户可以说"同时应用方案 1 和方案 3"，Agent 会合并调整。

### Q3: 调整后还是不够怎么办？

**A**: Agent 会再次生成调整建议，直到达到充足/可接受状态，或用户放弃。

### Q4: 预期增加百分比是如何计算的？

**A**: 基于经验值和统计数据：
- 粉丝数范围: 影响最大（50-150%）
- 新增粉丝: 中等影响（20-30%）
- 联盟达人: 中等影响（30-50%）
- 认证/账号类型: 较小影响（5-20%）

---

## 总结

Agent 的调整策略体系设计原则：

1. **智能评估**: 双重标准（真实数量 vs 保守估计）确保用户体验和系统稳定性
2. **优先级排序**: 优先推荐效果最好的调整方案
3. **透明沟通**: 明确告知当前值、新值、预期效果和理由
4. **用户主导**: 不自动调整，等待用户确认
5. **灵活迭代**: 支持多次调整，直到满足需求

这套策略确保了 Agent 在达人数量不足时，能够提供有价值的建议，同时保持对用户意图的尊重。
